How the GitHub integration works (AI Domain)

The AI Domain runs on a teacher-owned GitHub Organization. This document explains how data flows from GitHub into Forge — the tiers it passes through, what gets ingested, how it's attributed and rolled up into analytics, and where the security boundaries sit. GitHub is the source of truth; Forge is the layer on top.

1 · The three tiers

Every byte of GitHub data crosses these — and the browser never talks to GitHub.

GitHub
Organization, repos, issues, PRs, reviews, commitssource of truth
Where the work actually happens. Emits webhook events and answers read API calls.
Server
Express backend — webhook receiver, verifier, ingester, analyticsall logic + secrets
Verifies signatures, normalizes events, attributes them, stores them, exposes a clean scoped API. The only tier that holds GitHub credentials.
Client
Next.js portal — dashboards, workflow visualization, analyticsUI only
Calls the server's /api and nothing else. Renders the issue→PR→review→merge story. Holds no secrets, never calls GitHub.
Hard boundary: the client never reaches GitHub directly. All GitHub communication is server-side. The portal only ever sees Forge's own API.

2 · End-to-end flow

Two directions: GitHub pushes events in (webhooks), and the server pulls detail on demand (token).

# INBOUND — a student's work lands in the portal
Student raises PR  ──▶  GitHub Org
                          │  fires webhook (push / pull_request / pull_request_review / issues)
                          ▼  POST + X-Hub-Signature-256 (HMAC)
                      Express: /api/integrations/github/webhook
                          │  1. verify HMAC against GITHUB_WEBHOOK_SECRET   (reject if bad → 403)
                          │  2. normalize event → activity record
                          │  3. attribute: repo → team, github login → Forge user
                          │  4. upsert by externalId  (idempotent; redelivery-safe)
                          ▼
                      Database  (GithubActivity, rollups)
                          │
                          ▼  GET /api/integrations/github/activity  (scope-filtered)
                      Portal dashboards  ──▶  Teacher / Mentor / Mentee see it

# OUTBOUND — extra detail on demand (optional)
Server  ──▶  GitHub REST API  (read-only token, GITHUB_API_TOKEN)  ──▶  repo / issue / PR detail
Idempotent + verified. Each event is HMAC-verified and upserted by its GitHub externalId, so GitHub re-deliveries never create duplicates and a forged payload is rejected before it touches the database.

3 · What gets ingested

Only the four event types the AI workflow needs — the rest are safely ignored.

GitHub eventBecomesSurfaced as
pushone activity per commitcommit counts, activity feed, contributor analytics
pull_requestPR opened / merged / closedPR list, review queue, merge rate, issue fan-out
pull_request_reviewa review eventmentor review throughput, learning-eval anchor
issuesissue opened / closedopen/closed issue counts, the unit-of-work board

Pick “Send me everything” on the webhook if you prefer — the server verifies every delivery and simply no-ops on event types it doesn't model (returning 200). See github-setup.html for configuration.

How raw GitHub objects become the portal's domain model.

IssueIssue (unit of work)Students self-assign; one issue → many attempts.

The portal enhances, never duplicates. Issues/PRs/commits stay in GitHub. Forge stores enough to make them visible and mentorable — and adds what GitHub can't: the learning evaluation and cross-team analytics.

5 · Attribution & analytics rollups

Every event is tagged with a team and a person, then aggregated.

incoming event
   ├─ repo full-name        ──▶  Team        (repo belongs to one team)
   ├─ author github login   ──▶  Forge user  (profile-linked username)
   └─ type + state          ──▶  metric

rollups derived from the tagged events:
   Org      repos · teams · contributors · commits · PRs · open/closed issues
   Team     commits · PRs · merge-rate · issues closed · milestone progress
   Student  attempted · solved · PRs raised/merged · acceptance-rate · commits · review scores
   Mentor   reviews completed · PRs reviewed · team health

These power the Teacher comparison screens, the Mentor review queue, and the Mentee analytics — all from the same attributed event stream.

6 · Who can read what

The same events, filtered by role on every request.

RoleSees GitHub activity for…
Admin · LCCEvery domain, every team (global) — the LCC reads AI, ML and SDSE.
TeacherTheir own domain(s).
MentorTheir own team.
MenteeTheir own activity.
Read access is enforced server-side by the scope filter — the client receives only what the role is allowed to see. For multiple domain orgs, the LCC's cross-org read uses a classic read-only token owned by an all-orgs account (see github-setup.html).

7 · Security boundary

Webhook authenticity

Every delivery carries X-Hub-Signature-256; the server recomputes the HMAC over the raw body with GITHUB_WEBHOOK_SECRET and rejects mismatches with 403. Never skipped.

Secrets stay server-side

The webhook secret and read token live in server/.env / AWS Secrets Manager — never in the client bundle, never in git.

Client is sandboxed

The browser calls only Forge's /api. It cannot reach GitHub, cannot see tokens, and gets pre-filtered data.

Least privilege

The API token is read-only (Contents/Metadata/PRs/Issues). The portal never writes to GitHub.

8 · Mock now, real later

Phase 1 proves the experience; the wiring lands in Phase 2.

Phase 1 (current): the flow above is mocked end-to-end in the client — lib/mock/github.ts seeds a believable AI org and the dashboards render the full issue→PR→review→merge story. No real GitHub calls, no backend. This validates the product before any integration is built.
Phase 2+: the server-side webhook receiver, signature verification, attribution and scoped API are wired to the real org. Because the portal already reads through a single data-access seam (@/lib/api), flipping from mock fixtures to the live API is a one-place change — no screen rewrites.
