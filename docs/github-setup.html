0 Set up the AI org structure AI domain do this first

Steps 0–3 are the AI domain only (the GitHub-driven one). The AI dashboards are built around GitHub Teams (a mentor + their students, mapped to a repo). Create that structure once — it's all free-tier. ML/SDSE skip to Steps 4–5.

  1. You have the org newton-school-ai (free tier is fine — it supports Teams, webhooks and fine-grained-token policies).
  2. Create one GitHub Team per student team: Org → Teams → New team — e.g. team-alpha, team-beta, team-gamma. Add the mentor + the students as members of each.
  3. Create a repository per team and grant that team access (Team → Repositories → Add repository → Write). The repo URL is how Forge maps activity back to the team.
  4. Each member sets their GitHub username on their Forge profile so commits/PRs attribute to the right person.
Forge reads this structure through the org Teams API (needs the token's Members: Read-only permission, below). No GitHub Teams yet? The server can still read repos/issues/PRs, but team grouping & mentor attribution won't populate until Teams exist.

1 AI organization webhook AI domain required

One webhook at the org level covers every current and future AI repo — you don't add a webhook per repo. (ML/SDSE do the per-repo version in Step 5.)

  1. Open your org's webhook settings (owner only): github.com/organizations/your-org/settings/hooks/new — i.e. Org → Settings → Webhooks → Add webhook. (On a free org this lives under the organization settings, not a repo.)
  2. Payload URL — the server's webhook endpoint:
    Production
    Local devhttps://smee.io/<your-channel>
    GitHub can't reach localhost — webhooks POST from the public internet. For local dev, create a channel at smee.io/new, use that URL as the Payload URL, and forward it to your server:
    npx smee-client --url https://smee.io/<your-channel> --target http://localhost:4000/api/integrations/github/webhook
    (ngrok works too: ngrok http 4000 → https://<id>.ngrok.io/api/integrations/github/webhook.)
  3. Content type → application/json (required — the server reads JSON and verifies the raw body; x-www-form-urlencoded will not work).
  4. Secret — generate one and paste the same value into the webhook field and GITHUB_WEBHOOK_SECRET in server/.env:
    Generateopenssl rand -hex 32
    The server checks the X-Hub-Signature-256 HMAC on every delivery — without a matching secret it returns 403.
  5. SSL verification → Enable (production is HTTPS via the ALB). Leave disabled only for a plain-HTTP local tunnel.
  6. Which events would you like to trigger this webhook? → pick “Send me everything.” — the simplest choice. The server verifies the HMAC on every delivery and acts only on the four it understands (push · pull_request · pull_request_review · issues); any other event safely no-ops and returns 200. The only cost is slightly more inbound traffic.
    Prefer a leaner feed? Choose “Let me select individual events” instead and tick exactly: PushesPull requestsPull request reviewsIssues (untick the default “Just the push event”). Same result, fewer deliveries.
  7. Keep Active checked → Add webhook. GitHub immediately sends a ping; the server replies 200 and ignores it. Open Recent Deliveries and confirm a green ✓ 200 (re-deliver if you fix the secret).
  8. ML & SDSE don't repeat this. They have no org — instead they add a per-repo webhook (same URL, same secret) in Step 5 below — one click via the OAuth App (Step 4).

2 Read-only API token required for the AI dashboards

The server reads your org structure (repos, issues, PRs, reviews, milestones, Teams) through one GITHUB_API_TOKEN. A fine-grained token scoped to your org your-org is the right choice for a single org. (Free-tier orgs support all of this.)

Three real gotchas — get these right or you'll see a 403:
  1. Token lifetime must be ≤ 366 days. Many orgs (incl. this one) forbid fine-grained tokens with a longer lifetime — a 1-year+/no-expiry token is rejected with 403. Pick e.g. 90 days.
  2. Resource owner must be the org (not your personal account), which usually creates a pending approval an org owner must accept.
  3. Teams need Org → Members: Read-only. Without it, /orgs/<org>/teams returns "Resource not accessible" and team grouping won't sync.

Create the token (single org — recommended)

  1. New fine-grained token: github.com/settings/personal-access-tokens/new.
  2. Expiration → ≤ 366 days (e.g. 90 days). This is the #1 cause of a 403 — don't pick "No expiration".
  3. Resource owner → your-org (the organization, not your user). If it shows “request will require approval”, an org owner approves it under Org → Settings → Personal access tokens → Pending requests.
  4. Repository access → All repositories (covers new team repos automatically).
  5. Repository permissions — all Read-only:
    PermissionAccess
    ContentsRead-only
    MetadataRead-only (auto-selected)
    IssuesRead-only
    Pull requestsRead-only
  6. Organization permissions → Members: Read-only. Required to read GitHub Teams (mentor + students per repo).
  7. Generate token → copy the github_pat_… value into server/.env as GITHUB_API_TOKEN, and set GITHUB_ORG to your org slug your-org.
Verify (no server needed)
node --env-file=.env -e 'fetch(`https://api.github.com/orgs/${process.env.GITHUB_ORG}`,{headers:{Authorization:`Bearer ${process.env.GITHUB_API_TOKEN}`,"User-Agent":"forge"}}).then(r=>r.json()).then(o=>console.log(o.login||o.message))'
Run it in portal/server. It should print your org login — if it prints a 403 message about token lifetime, fix the expiration (gotcha #1).
One org, one token. Only the AI domain has a GitHub org (newton-school-ai); ML/SDSE connect individual repos (Step 5) and read them as public repos, so this single fine-grained token covers everything. (If a domain ever gets its own GitHub org, a GitHub App is the multi-org path — not needed today.)

3 Link members to their GitHub all domains

So activity shows up against the right person in Forge — applies to AI, ML and SDSE alike.

  1. Each member clicks Connect with GitHub in the Connections tab. The OAuth App (Step 4) returns their verified username — read from GitHub, not typed, so no one can claim someone else's handle. No per-user token is stored.
  2. AI: the same handle is a member of the org, so their commits/PRs flow through the org webhook. ML/SDSE: their commits/PRs flow through their team's repo webhook (Step 5) — no org membership needed.
  3. Forge resolves each event's author by username and attributes it to that user and their team (matched by the team's repo). Unmatched activity is still recorded against the team.
One OAuth App backs the Connect button — but no GitHub App, no installation id, no private-key .pem, and no per-user tokens kept on the server (the mentor's token is used once to create a webhook, then discarded). One webhook secret + one read-only token + the OAuth App is the whole integration.

4 One-click connect — the OAuth App all domains (register once)

A single GitHub OAuth App, owned by your dedicated lcc-ai-nst account, powers the Connect with GitHub button everywhere — it verifies usernames and lets a mentor's one click auto-create the ML/SDSE repo webhook (no manual Settings → Webhooks). An OAuth App ≠ the org GitHub App we don't use — it's just "Sign in with GitHub", no private key, no installation.

  1. Sign in as lcc-ai-nst → Settings → Developer settings → OAuth Apps → New OAuth App.
  2. Application name → Forge. Homepage URL → https://app.example.com.
  3. Authorization callback URL — the server's callback:
    Production
    Local devhttp://localhost:4000/api/integrations/github/oauth/callback
  4. Register → copy the Client ID, then Generate a new client secret and copy it. These go in server/.env (below). The secret is shown once — store it now.

5 ML & SDSE — connect a repo ML · SDSE (no org needed)

ML and SDSE run the native Forge workflow and have no GitHub org. With the OAuth App registered (Step 4), connecting a team's repo is one click + a URL — done by the team's mentor / lead from the Connections tab. Students just link their username.

  1. Every member → Forge → Connections → Connect with GitHub. Verifies their username (attributes their commits/PRs). That's all students do.
  2. The mentor → same Connect with GitHub, then paste the team repo URL (https://github.com/owner/repo or owner/repo) → Submit.
  3. Forge does the rest — using the mentor's authorization it creates the webhook on that repo automatically (POST /repos/{owner}/{repo}/hooks with the payload URL, the shared GITHUB_WEBHOOK_SECRET, and events push pull_request pull_request_review issues), stores the repo against the team, and confirms a green ✓. No Settings → Webhooks, no copy-pasting the secret.
  4. Reads — keep the repo public and the read token sees it with nothing extra; for a private repo, add lcc-ai-nst as a read-only collaborator so dashboards can pull detail.
Forge needs admin on the repo to create the webhook — the mentor has it, so their one-time authorization is enough. If a mentor would rather not grant admin:repo_hook, they can still add the webhook by hand: repo → Settings → Webhooks → Add webhook with the same payload URL, application/json, the shared secret, and the four events.
Why this works without an org. The server verifies every delivery against the one GITHUB_WEBHOOK_SECRET and attributes each event by repo URL + author username — it never needs an org. A repo webhook is just the org webhook scoped to a single repo.
