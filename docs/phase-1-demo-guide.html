Forge — the platform, end to end

A guided tour of the Profile Building Drive platform built by the Learner Career Council at Newton School of Technology. This is a presentation build: every screen is real, the data is mocked, and you can switch role and domain to see exactly how the platform behaves before any backend is built.

1 Platform overview

Forge runs a multi-domain student drive: students build, mentors guide, teachers track, and learning becomes measurable. GitHub is integrated two different ways depending on how a domain actually works.

Roles

Admin · LCC · Teacher · Mentor · Mentee. (Team Lead is an alias of Mentor.)

Domains

AI · ML · DVA · SDSE — each a cohort with its own teams, mentors and repositories.

Integrations

GitHub, Discord, Google Calendar, Email, and an AI assistant layer.

Two GitHub modes power the experience: Organization for AI, and Repository for ML/DVA/SDSE — both feed the same dashboards.

2 How to demo (presentation mode)

In the top bar there are two switchers. Use them to jump across the entire platform instantly.

Viewing as Role

Switch between Admin, LCC, Teacher, Mentor, Team Lead (opens the Mentor view), and Mentee. Each role's whole sidebar + dashboards change.

Domain AI · ML · DVA · SDSE

Switch the active domain. AI shows the organization-driven GitHub experience; ML/DVA/SDSE show the repository-based experience. The sidebar's GitHub section changes to match.

Suggested walk: Mentee on AI (org issues → PRs → reviews) → switch domain to ML (repository: collaborators, branches, releases, contributions) → switch to DVA (same, but Issues optional — see the empty state) → switch role to Mentor then Teacher.

4 AI domain — organization flow Organization

AI is GitHub-first. A teacher-owned organization is the source of truth and the workflow lives on GitHub.

Organization→Teams→Repositories→Issues→Pull Requests→Reviews→Milestones

Issues are the unit of work; multiple students may attempt one issue (each a PR); the mentor reviews the PR and evaluates learning (understanding · explanation · technical depth) before merging — which advances the milestone.

5 ML / DVA / SDSE — repository flow Repository

These domains have no organization. A Team Lead or Mentor owns one repository and students are collaborators — the repository is the source of truth, and issues are optional.

6 GitHub integration flow

How each mode connects (the demo simulates this; real sync is a later phase).

Organization AI

  • One org webhook covers every repo
  • One read token reads org structure + Teams
  • Activity attributed by team (repo) + author (login)

Repository ML / DVA / SDSE

  • Mentor/Lead clicks Connect with GitHub + pastes the repo URL
  • Server auto-creates the repo webhook (one click)
  • Students link their username so contributions attribute
This is a presentation build. No real GitHub sync runs — every dashboard is filled with realistic mock data so the experience and workflows can be reviewed first. The permission analysis for real repository sync is in github-integration-review.md.

7 Role-by-role experience

Admin

User management, invitations, domains, teams, connections, configuration, audit logs, email testing.

LCC

User onboarding (invite-only), communication/email center, team tracking, domain analytics, drive health.

Teacher

Domain dashboard, team analytics, L4 reviews — plus repository analytics across the domain (org or repo mode).

Mentor

Team dashboard, mentee tracking, L3 reviews, deliverables — plus the team repository: health, contributors, branches, PRs, student performance.

Mentee

My repository, issues (if used), my pull requests, contribution analytics, milestones, daily updates and feedback.

Team Lead

An alias of Mentor — opens the Mentor dashboards. In repository mode a Team Lead may own the repo (shown in the repository view).
