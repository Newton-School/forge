Authentication & onboarding — how access works

Forge is a private, invite-only platform. Accounts are created by Admin or LCC; users sign in with Google only. Being authenticated by Google never grants access on its own — the email must already exist in the database and pass every check below.

The onboarding journey

From account creation to first sign-in. No signup page, no registration, no self-onboarding.

Server

Validate access

Server

Grant access

Session created, role loaded, status → Active. Dashboard opens.

✓ Allowed — email is provisioned

The Google email exists in the users table, the account is Active/Invited and not suspended, and it holds a role. A server-side session (Redis) is created and the user enters the portal. An Invited user flips to Active on this first successful sign-in.

✕ Denied — not provisioned

The email isn't in the database, the hosted domain isn't rishihood.edu.in, or the account is suspended/disabled. No session is created. The user sees an access-denied message: “That account isn't allowed. Use your institute Google account that's already been added to Forge.”

Examples — Allowed: taj@rishihood.edu.in, mentor@rishihood.edu.in (already in the DB).   Rejected: random@gmail.com, unknown@rishihood.edu.in (authenticates with Google, but isn't provisioned).

Access validation — every login

All five must pass on every sign-in. Google authentication alone is never sufficient.

  • 1
    Email exists in the databaseAdmin-provisioned allowlist — unknown emails are rejected.
  • 2
    Hosted domain is the institute domainGoogle hd / email domain must be rishihood.edu.in.
  • 3
    Account is active, not suspendedSuspended / Disabled accounts are blocked.
  • 4
    User has an assigned roleRole + scope load into the session for RBAC.
  • 5
    User has portal accessOnly then is a server-side session created.
Defense in depth. These run server-side on the OAuth callback (in resolveGoogleLogin). The browser only ever holds an opaque, HttpOnly session id — no tokens, no role logic the client can tamper with.

Invitation lifecycle

Each created user gets one tracked invitation. Admin/LCC can watch its status and resend.

Pending→ Sent→ Opened→ Completed | Expired
Pending created, not yet sent  ·  Sent email dispatched (sentAt)  ·  Opened tracking pixel hit (openedAt)  ·  Completed user signed in with Google (acceptedAt)  ·  Expired link passed expiresAt before completion (resend to reissue).
