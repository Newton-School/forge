AI Domain — Portal Guide & User Journey

Forge is a multi-domain university Profile Building Drive platform. For the AI Domain, work happens on a teacher-owned GitHub Organization — so the portal is the visibility, mentoring, evaluation and governance layer on top of GitHub, not a second place to track work. This guide explains the structure, the user journeys, the navigation, and the GitHub workflow the product is built around.

1 · Platform overview

What the portal is, and why it exists.

Forge gives students, mentors, teachers, the LCC and admins one system to run a multi-week skill-building drive across three domains — AI, ML and SDSE. Each domain can run a different workflow.

Core principle (AI Domain): GitHub is the source of truth. The portal should not duplicate GitHub — it should enhance it: surface repos/issues/PRs, make the mentor review & learning evaluation first-class, and roll activity up into analytics teachers and the LCC can act on.

What the portal adds

Visibility (who's doing what), mentoring (PR review + learning evaluation), analytics (team/student/mentor rollups), and governance (comparison, oversight, escalation).

What it never does

Re-implement issues, PRs or commits. Those live in GitHub. The portal reads them (via the backend in Phase 2) and layers meaning on top.

2 · AI Domain structure

Organization → Teams → Repositories → Issues → Pull Requests → Reviews.

Organization (teacher-owned)
├── Team Alpha     mentor + students   →  repo: resume-intel
├── Team Beta      mentor + students   →  repo: forecast-lab
└── Team Gamma     mentor + students   →  repo: engage-iq

Repository
├── README · Dev Guide · Contributing · POD Guide · Setup
├── Milestones        (progress advances as PRs merge)
├── Issues            (the unit of work)
│     └── Pull Requests   (one issue → many student PRs)
│            └── Mentor Review   (code decision + learning evaluation)
└── Commits · Contributors · Activity

A project may be assigned to one or many teams — never assume one team per project. The internal hackathon, for example, spans all three teams.

EntityMeaning
OrganizationThe teacher owns it; everything lives inside.
TeamA mentor + their students + the team's repository.
RepositoryWhere the work is; surfaced with its docs, milestones, issues and PRs.
IssueThe unit of work. Students self-assign and attempt it.
Pull RequestA proposed solution. Many students may raise PRs for the same issue.
ReviewThe mentor's decision and a learning evaluation, kept historically.

3 · The GitHub workflow

Every contribution follows the same path — and the portal visualizes it everywhere.

→

→

→

Because progress is tied to merged PRs, milestone completion is real, not self-reported. This pipeline is rendered on the issue detail (with the current step highlighted), the mentee dashboard, and a dedicated /github/workflow page.

4 · Multiple mentees, one issue

The reality the platform models — not "one issue, one student".

Issue #42  "Implement résumé section parser"
├── Sneha   PR #101   MERGED  ✓  layout-heuristic approach
├── Rohan   PR #102   REJECTED  regex breaks on 2-column; couldn't explain failure
├── Aniket  PR #103   REJECTED  no tests; couldn't justify heading detection
└── Priya            ATTEMPTING  branch open, no PR yet

The portal visualizes who attempted, who raised PRs, which PR merged, and which were rejected and why. The mentor chooses the PR that gets merged — and the rejections carry a reason so they're a learning moment, not a dead end.

5 · Learning-oriented evaluation

Mentors don't only grade code — they assess whether a student can explain it.

On every PR review, the mentor records three 1–5 scores plus notes, stored historically:

DimensionQuestion the mentor answers
UnderstandingDo they grasp why their approach works?
ExplanationCan they articulate it clearly to others?
Technical depthHow sound is the engineering underneath?

Example notes captured in the portal: "Can explain the layout heuristic and why it beats regex." · "Could not explain multi-column failure — pair on document structure."

6 · User flows

What each role does, start to finish.

Teacher

Login → Org Dashboard
   → Team Comparison (A vs B vs C)
   → Project Comparison (multi-team)
   → Repository analytics
   → Mentor performance + Student tracking
Complete visibility: compare teams and projects on commits, PRs, merge rate, issues closed and milestone progress.

Mentor (Team Lead)

Login → Team Dashboard
   → Review queue (open PRs)
   → Open a PR → review + learning eval
   → Merge best PR / reject with reasons
   → Track student contributions
The mentor is the team lead — mentorship and delivery in one role.

Mentee

Login → My Repository
   → Pick an Issue (self-assign)
   → Branch → Develop → Raise PR
   → Get mentor feedback
   → Watch analytics + milestones
Sees their attempts, PR outcomes, acceptance rate and the mentor's learning evaluation history.

Admin / LCC

Login → AI Domain Overview
   → Org analytics (repos/teams/PRs/issues)
   → Team comparison + activity
   → (Admin) user & domain management
The LCC reads every domain (global scope); teachers see their domain; mentors their team.

7 · Navigation

Domain-aware sidebar — AI adds a GitHub section; ML/SDSE keep the drive workflow.

Sidebar  =  role nav  +  (domain == AI ? GitHub nav : nothing)

Mentor · AI domain
├── Mentorship      Dashboard · My Mentees · Reviews
├── Team Delivery   Team · Issues & PRs · Blockers · Tasks · Deliverables
└── GitHub · AI     Team Dashboard · Repository · Issues · Pull Requests · Students   <- AI only

Mentor · ML / SDSE domain
├── Mentorship      Dashboard · My Mentees · Reviews
└── Team Delivery   Team · Issues & PRs · Blockers · Tasks · Deliverables            (no GitHub section)

A top-bar Domain switcher (AI / ML / SDSE) and Viewing-as role switcher (incl. Team Lead → Mentor) let stakeholders preview any combination. Switching the domain re-composes the navigation, dashboards and metrics. Shared detail pages — /github/repos/[repo], /github/issues/[id], /github/pulls/[id], /github/workflow — are reachable from any GitHub screen.

Each repository maps to a team (and a team to its mentor). A project can pull in several teams; GitHub Teams will become the source of truth for membership and permissions in a later phase.
