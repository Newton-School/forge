1Overview

Portal is an internal platform with a clean split between presentation and logic: a Next.js client renders the UI, and an Express server owns all business logic, data access, and integrations.

Traffic enters through Cloudflare (WAF, DDoS protection, DNS proxy), resolves via Route 53, and terminates TLS at an Application Load Balancer (certificate from ACM). The ALB fronts an ECS Fargate cluster running two containers — the Next.js SSR client on port 3000 and the Express API on port 4000 — backed by RDS PostgreSQL. Supporting services include ECR (images), Secrets Manager (config/secrets), CloudWatch (logs/metrics), and an optional ElastiCache Redis. Everything runs inside a dedicated VPC with public and private subnets and a NAT gateway. The ALB routes /api/* to the server and everything else to the client; the client never touches the database or third-party APIs directly.

2Repository structure

One repository, portal/, containing two independent applications plus shared tooling and docs.

This is deliberately not a Turborepo and has no apps/ or packages/ layout. There is no shared compiled package and no monorepo build orchestration to maintain. The two apps are decoupled and communicate only over the HTTP API contract, which keeps the mental model simple: the client is a UI app, the server is an API app, and they are deployed as separate containers. A build tool like Turborepo earns its keep when many packages share code and build graphs — here, with exactly two services and a clean network boundary between them, it would add ceremony without payoff.

3Client application

client/ — Next.js 15 (App Router), TypeScript, Tailwind, shadcn/ui. UI only

The client is responsible for rendering and interaction: routing, layouts, server-side rendering of pages, forms, and presentational components. It holds no business logic. It does not connect to the database and does not call GitHub, Discord, Google, or Groq directly.

How it talks to the server

  • All data flows through the HTTP API exposed by the server under /api/*.
  • The ALB routes /api/* requests to the server container; everything else is served by the client.
  • Session state rides on a secure, httpOnly cookie set by the server — the client never reads or constructs auth tokens.
  • Because authorization is enforced server-side, the client may hide UI it can't use, but it can never grant access the server would deny.

4Server application

server/ — Node.js, Express, TypeScript. All business logic

The server uses a conventional layered architecture so responsibilities stay isolated and testable:

  • Request logging and correlation
  • Session resolution (httpOnly cookie → user)
  • Authentication guard (rejects unauthenticated requests)

5Shared concerns

Types and validation are shared through the API contract, not a compiled package.

The boundary between client and server is the HTTP API. Request and response shapes, plus validation rules, are defined and enforced on the server; the client consumes the same contract. With only two apps and a clean network boundary, there is no shared package and none is needed today.

Future option

A shared packages/types (or generated client from an OpenAPI/Zod schema) could be added later if duplication grows. It is intentionally deferred to keep the current two-app model lightweight.

6Docker architecture

Two multi-stage Dockerfiles (one per app) plus a docker-compose for local development.

Each image builds in stages — install & build in a fat builder stage, then copy only the runtime artifacts into a slim final image. Both images run as a non-root user and are hardened (minimal base, no dev dependencies in the final layer).

7ECS deployment flow

Build → push to ECR → register task definition → ECS rolling deploy.

  1. Build — CI builds the client and server multi-stage images.
  2. Push to ECR — both images are tagged and pushed to Amazon ECR.
  3. Register task definition — a new ECS task definition revision references the new image tags, with secrets injected from Secrets Manager and logs wired to CloudWatch.
  4. Rolling deploy — ECS Fargate starts new tasks, waits for ALB target health checks to pass, then drains and stops the old tasks. No downtime.
CI build ──► ECR ──► Task Def revision ──► ECS service update │ │ │ │ ▼ ▼ ▼ ▼ client+ image secrets from rolling: start new → server tags Secrets Manager health check (ALB) → images + CloudWatch logs drain → stop old

8Authentication flow

Google OAuth (OIDC) only. No passwords No self-signup

There is no email/password login and no signup path. Access is granted by a two-part gate: the Google account must belong to the expected hosted domain (hd, e.g. rishihood.edu.in) and the email must already exist in the users table (allowlist). Unknown emails are rejected outright. Authenticated users get a server-side session carried by a secure, httpOnly cookie.

  1. User clicks "Sign in with Google" in the client.
  2. Server redirects to Google's OIDC consent.
  3. Google authenticates the user and returns an authorization code.
  4. Server exchanges the code for ID/access tokens and verifies them.
  5. Server checks the hd claim matches the allowed hosted domain.
  6. Server confirms the email pre-exists in the users allowlist table.
  7. If both pass, server creates a session and sets a secure httpOnly cookie; otherwise the request is rejected.

9RBAC flow

Authorization = role × scope, enforced server-side only.

Admin LCC Teacher Mentor Mentee

Every permission is the combination of a role (five total) and a scope that bounds how far that role's access reaches.

ScopeReach
GLOBALAcross the entire platform
DOMAINWithin a domain / cohort
TEAMWithin a single team
SELFOnly the user's own records

Config note — "Team Lead"

The refactor prompt re-listed a Team Lead role, but it was previously merged into Mentor — the Student Mentor leads the team. The current model has five roles. Re-adding Team Lead as a distinct role is a small config change if confirmed.

Three layers of server-side enforcement

  1. Route middleware — rejects requests whose role can't reach the endpoint at all.
  2. Service checks — applies role × scope rules to the specific action and resource.
  3. Data scoping — repository queries are filtered to the caller's scope (e.g. team or self) so out-of-scope rows are never returned.
Request (session → user: role, scope) │ ▼ Layer 1 — route middleware: role allowed on this route? ──► no → 403 ▼ Layer 2 — service: role × scope allows action? ──► no → 403 ▼ Layer 3 — repository: query filtered to scope (team/self) ▼ Response (only in-scope data)

The client may hide controls it can't use, but enforcement is never trusted to the client — all three layers run on the server.

10GitHub integration

GitHub App + webhooks, handled server-side only.

  • A GitHub App grants scoped access; webhooks push repository events to the server.
  • The server tracks engineering activity (e.g. commits, pull requests, and related contribution signals) to attribute work and surface it in the UI.
  • All credentials and webhook handling live on the server — the client only renders the resulting data via /api/*.

11Discord integration

Bot + OAuth verification + announcements.

  • Bot — operates in the community server for automation and messaging.
  • OAuth verify — links a user's Discord identity to their Portal account.
  • Announcements — the server posts updates/notifications into Discord channels.

As with all integrations, tokens and API calls are server-side only.

12Calendar integration

Google Calendar event sync.

  • The server syncs events with Google Calendar so schedules stay consistent between Portal and Google.
  • Calendar credentials are managed server-side; the client renders synced events through the API.

13Security & AWS account isolation

The AWS account is shared with other services — Portal is fully isolated so there is zero blast radius into them. Isolated

Because the same account also runs unrelated services, Portal is provisioned with its own dedicated boundary at every layer. A compromise or misconfiguration in Portal cannot reach the other services, and vice versa.

LayerIsolation
NetworkDedicated VPC, public + private subnets, NAT; no peering to other services' networks
Access controlDedicated security groups and IAM roles/policies scoped to least privilege
SecretsDedicated Secrets Manager entries and KMS keys
Data & transportPrivate-subnet RDS; TLS at the ALB (ACM); secure httpOnly session cookies; non-root hardened containers
Blast-radius containment

Least-privilege IAM, no network peering, and dedicated VPC / SG / Secrets / KMS mean Portal is sealed off from the account's other services. There is no lateral path in or out.

14Getting started

Clone, run the stack with Docker Compose, and start the apps locally.

git clone <repo-url> portal cd portal docker-compose up # client + server + postgres (+ redis if enabled)
