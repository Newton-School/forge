How Forge runs on AWS

A plain-English tour of the production setup — what each piece does, how a request flows, how we ship changes, and how we stay secure and reliable. No AWS background needed.

Forge runs as two containers — the UI and the brain — behind a single web address, with a managed database and cache, on Amazon's managed container platform (ECS Fargate). It scales, heals itself, ships automatically, and emails us the moment something looks wrong.

The 20-second version Users hit one address (forge.taj.works). Cloudflare shields it, a load balancer sends page requests to the frontend and /api requests to the backend. The backend holds all the logic and talks to the database (Postgres), the session store (Redis), and outside services (Google, GitHub, Discord). Everything sensitive is locked in a private network; nothing but Cloudflare can reach it.

What happens on a single request

  1. A user opens forge.taj.works. The request first hits Cloudflare, which blocks bad traffic, serves static assets fast, and forwards the rest.
  2. Cloudflare passes it to the Load Balancer over an encrypted connection. The load balancer is the only door into our AWS network.
  3. The load balancer decides where it goes by the URL: anything starting with /api/ goes to the backend; every other page goes to the frontend. Same address, one origin — so logins and cookies "just work".
  4. The frontend renders the page and asks the backend for data through the same /api path.
  5. The backend does the real work: checks your session in Redis, confirms you're allowed to see it, reads/writes Postgres, and calls Google/GitHub/Discord when needed.
  6. The answer travels back up the chain to the user — all within a private, encrypted network.

Why two apps instead of one

Frontend — “the UI”Backend — “the brain”
Only draws screens. Never touches the database, never holds a secret, never calls GitHub/Google directly. Owns everything that matters: authentication, who-can-see-what, business rules, integrations, email, audit logs.
If it's ever compromised, there's nothing sensitive to steal. Locked in the private network; the only thing that enforces permissions. The frontend's UI checks are just hints.

This separation is the core security decision: trust is enforced in one place, not scattered across the browser.

How a change goes live

Shipping is automated — a developer merges code and the pipeline does the rest, with a safety net.

  1. Code is merged (or the deploy is triggered manually).
  2. GitHub Actions logs into AWS securely using short-lived trust — no passwords or keys stored anywhere.
  3. It builds both apps into images and stores them in ECR.
  4. It updates the database structure safely, as a one-off task inside the private network.
  5. It rolls out the new version gradually. Each new container must pass a health check before old ones retire.
  6. If the new version is unhealthy, it automatically rolls back to the last good one — users never see a broken deploy.

Reliability & “what if something breaks?”

  • Two copies of everything, in different data centers (availability zones) — one failing doesn't take the app down.
  • Self-healing — an unhealthy container is automatically replaced.
  • Safe deploys — a bad release auto-rolls-back.
  • Persistent sessions — restarts don't log users out.
  • We get emailed the moment there's trouble: app errors, crash loops, error spikes, database/cache pressure, unusual AWS activity, or a failed deploy — all to learnercareercouncil@nst.rishihood.edu.in.

Scaling & cost posture

  • Small and cheap to start — modest database/cache sizes; a single shared internet gateway to keep costs down.
  • Scales sideways — under load we simply run more copies of a container; the load balancer spreads traffic automatically.
  • Managed, not hand-run — AWS handles the servers, patching, and backups, so the team focuses on the product, not infrastructure.
Bottom line for the product team Forge is on a standard, boring-in-a-good-way production setup: one address, a shielded and private network, a self-healing container platform, automatic deploys with rollback, persistent data, full audit trails, and email alerts. It's built to grow with the drive and to fail safely.
