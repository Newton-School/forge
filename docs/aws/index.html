Production Runbook

Deploying Forge on AWS (ECS Fargate)

Step-by-step deployment of the split client (Next.js SSR) + server (Express API) platform onto AWS, per CLAUDE.md & docs/infra-ecs.md. Companion artifacts live in infra/terraform/, infra/ecs/, infra/aws/, docker/, and .github/workflows/deploy.yml.

Why AWS fixes the email problem for free Unlike Render's free tier (which blocks outbound SMTP 25/465/587), AWS ECS/Fargate leaves 587/465/2525 open — only port 25 is throttled. So Gmail SMTP on 587 connects fine here, and the natural choice is Amazon SES (cheap, high limits, native). See Phase 7.
This is now real Terraform — the phases below are the manual/click-ops equivalent infra/terraform/ provisions everything in these phases (VPC · NAT · SGs · KMS · RDS · Redis · ECR · Secrets · IAM + the GitHub OIDC deploy role · ACM+Cloudflare · ALB path routing · ECS cluster/services · CloudWatch alarms + SNS email) wired by reference, terraform validate ✓. The phases below stay as the conceptual map (and the by-hand path). Fastest route to production: 
# 0. one-time state backend — VERSIONED S3 + DynamoDB lock (separate root module)
cd infra/terraform/bootstrap && terraform init && terraform apply
# 1. init the main module against that backend, then apply
cd .. && terraform init -backend-config=... && terraform apply
Full steps in infra/terraform/README.md. After apply: fill the external secrets, confirm the SNS alert email, and copy the outputs into the GitHub Actions vars (Phase 12).

Phase 0 · Prerequisites ~30 min

  • An AWS account with admin access (or a dedicated Forge account — recommended for blast-radius isolation per CLAUDE.md §6).
  • AWS CLI v2 configured: aws configure (set default region, e.g. ap-south-1).
  • Docker installed locally (for the first manual image build).
  • Terraform ≥ 1.6 — infra/terraform/ is a complete, validated module that provisions every phase below. Strongly preferred over click-ops.
  • Domain in Cloudflare (taj.works) — you already have this. For automated DNS, a scoped CLOUDFLARE_API_TOKEN (DNS:Edit) + the zone ID.
  • Pick the production app tag used everywhere: app=forge, env=prod.
One-time: the Terraform state backend (versioned S3 + DynamoDB lock) Remote state lives in an S3 bucket with versioning on (recover from a bad apply), KMS encryption, and full public-access block; locking uses a DynamoDB table. It's a separate root module (infra/terraform/bootstrap/) with local state — the bucket that holds state can't hold the state of the run that creates it. Run it once before the main module.
Single ALB, path-based routing (the AWS-canonical topology) On AWS the recommended layout is one domain (e.g. forge.taj.works) behind one ALB that routes /api/* → server and everything else → client. This removes the cross-subdomain cookie complexity you hit on Render/Vercel — the browser sees one origin, so SameSite=Lax first-party cookies "just work" and you can drop COOKIE_DOMAIN.

Phase 1 · Network & account isolation

Provision a dedicated VPC — never share with other apps (CLAUDE.md §6).

  • VPC with 2 AZs. Public subnets (ALB + NAT GW) and private subnets (ECS tasks, RDS, Redis).
  • 1 NAT Gateway per AZ (or 1 to save cost) so private tasks have IPv4 egress (this is what makes SMTP/Google/GitHub calls work).
  • Security groups (least-privilege):
    • sg-alb ← inbound 443 from Cloudflare IP ranges only.
    • sg-ecs ← inbound from sg-alb on 3000 & 8000.
    • sg-rds ← inbound 5432 from sg-ecs only.
    • sg-redis ← inbound 6379 from sg-ecs only.

All of this is in infra/terraform/network.tf + security_groups.tf (with the ALB SG locked to Cloudflare's published IPv4 ranges). Run terraform plan and read it before apply.

Phase 2 · Data — RDS Postgres + ElastiCache Redis

RDS for PostgreSQL

  • Engine: PostgreSQL 16. Instance: start db.t4g.micro (scale later).
  • Not publicly accessible; private subnets only; sg-rds.
  • Storage encrypted with the Forge KMS CMK. Automated backups on.
  • Create the database + app user. Capture the connection string:
# Prisma runtime (pooled) + direct (for db push). sslmode=require is mandatory.
DATABASE_URL=postgresql://forge:<pw>@<rds-endpoint>:5432/forge?sslmode=require
DIRECT_URL=postgresql://forge:<pw>@<rds-endpoint>:5432/forge?sslmode=require
This solves your session-eviction pain RDS + ElastiCache are persistent, so sessions and data survive restarts — no more intermittent 401s from Render's ephemeral free Redis.

ElastiCache Redis

  • Redis (cluster mode disabled is fine to start), 1 node, private subnets, sg-redis.
  • Capture REDIS_URL=redis://<primary-endpoint>:6379 (used for server-side sessions).

Phase 4 · ECR + build & push images

Create two ECR repos and push the first images manually (CI takes over afterwards).

aws ecr create-repository --repository-name forge/client
aws ecr create-repository --repository-name forge/server

# authenticate docker to ECR
aws ecr get-login-password --region $REGION | docker login --username AWS --password-stdin $ACCT.dkr.ecr.$REGION.amazonaws.com

# build (Dockerfiles already exist in docker/)
docker build -f docker/server.Dockerfile -t $ACCT.dkr.ecr.$REGION.amazonaws.com/forge/server:latest portal/server
docker build -f docker/client.Dockerfile -t $ACCT.dkr.ecr.$REGION.amazonaws.com/forge/client:latest portal/client \
  --build-arg NEXT_PUBLIC_API_URL=/api --build-arg APP_MODE=production

docker push $ACCT.dkr.ecr.$REGION.amazonaws.com/forge/server:latest
docker push $ACCT.dkr.ecr.$REGION.amazonaws.com/forge/client:latest

The client builds with output: "standalone" and is configured at build time with NEXT_PUBLIC_API_URL=/api + APP_MODE=production (it's UI-only — no secrets).

Phase 5 · IAM roles

Two roles per the JSON in infra/aws/:

  • Execution role (iam-ecs-execution-role.json) — lets ECS pull from ECR, write CloudWatch logs, and read the /forge/prod/* secrets.
  • Server task role (iam-server-task-role.json) — the app's own runtime permissions (e.g. SES ses:SendEmail if you use SES, scoped to your verified identity).

Scope every policy to Forge ARNs + aws:ResourceTag/app = forge; no wildcard Resource: "*" (the only allowed exception is ecr:GetAuthorizationToken).

Phase 6 · ALB + ACM (TLS) + routing

  • Request an ACM certificate for forge.taj.works (DNS-validated).
  • Create an Application Load Balancer (internet-facing, public subnets, sg-alb).
  • Two target groups: tg-client (port 3000) and tg-server (port 8000), both ip target type.
  • HTTPS:443 listener (ACM cert). Rules:
    • /api/* → tg-server
    • default → tg-client
  • Health checks: server → GET /api/health (200); client → / (200).
Cookie simplification Because client & server are now the same origin (forge.taj.works), set COOKIE_DOMAIN empty (host-only cookies) and the client's API_PROXY_TARGET can be the internal ALB / http://localhost:8000-equivalent. The cross-subdomain CSRF/session workarounds you needed on Render are no longer required.

Phase 7 · Email — Amazon SES (recommended) or Gmail

SMTP egress works on AWS, so you have two clean options. No application code changes — only env/secrets differ (the nodemailer transport already handles any host/port + IPv4 resolution).

Phase 8 · ECS task definitions + services

  • Create the ECS Fargate cluster (forge-prod).
  • Register the two task defs from infra/ecs/task-def.client.json and task-def.server.json — fill in the rendered image URIs, the execution/task role ARNs, the /forge/prod/* secret ARNs, and the log group.

  • Create two ECS services (client → tg-client, server → tg-server), desired count 2 each, in private subnets with sg-ecs. Enable the deployment circuit breaker (auto-rollback).

Phase 9 · DNS — Route 53 + Cloudflare

  • In Cloudflare, point forge.taj.works (CNAME) at the ALB DNS name.
  • Proxy status: proxied (orange cloud) so Cloudflare WAF/DDoS sits in front, per the topology.
  • SSL/TLS mode: Full (strict) — Cloudflare ↔ ALB over the ACM cert.
  • Lock sg-alb inbound 443 to Cloudflare IP ranges only, so nobody bypasses the proxy.

Phase 10 · OAuth redirect URIs & app config

  • Google Cloud Console → Credentials → your OAuth client → Authorized redirect URIs: add https://forge.taj.works/api/auth/google/callback. Authorized JS origins: https://forge.taj.works.
  • GitHub OAuth App (if used) → callback URL → https://forge.taj.works/api/integrations/github/oauth/callback.
  • Webhooks (GitHub/Discord) → point at https://forge.taj.works/api/integrations/.../webhook.

Phase 11 · Database schema (Prisma) + seed

Forge uses prisma db push (not migrations). Run it once against RDS — from a one-off ECS task, a bastion, or locally through a tunnel using DIRECT_URL:

cd portal/server
npx prisma generate
npx prisma db push          # creates the schema on RDS
npm run db:seed             # seeds roles / allowlisted users
Allowlist is the gate Google login only succeeds if the email's hd is rishihood.edu.in and the user already exists in the DB. Seed (or admin-provision) the initial admin before first login, or nobody can get in.

Phase 12 · CI/CD — GitHub Actions OIDC

What this file is. .github/workflows/deploy.yml is the deployment pipeline for Forge — the single automation that turns a commit on main into a running production release on ECS. It authenticates to AWS with GitHub OIDC (a short-lived token, no static AWS keys in the repo), builds both app images, syncs the database schema, and performs a zero-downtime rolling deploy. After it's wired up, shipping is just: merge to main → CI ships it — the same trigger model you used on Render/Vercel.

What it does, in order. The workflow runs three dependent jobs:

  • 1 · terraform — converges infrastructure with terraform apply (idempotent — a no-op when nothing changed). It auto-detects whether infra/terraform/*.tf exists: if the infra is already provisioned (or you set skip_terraform), it skips straight to deploy. So it handles both "stand up the infra" and "infra is ready — just deploy".
  • 2 · build — assumes the AWS deploy role, ensures the two ECR repos exist (forge-server, forge-client), then builds both Dockerfiles and pushes images tagged with the commit SHA (plus latest). The client image is built with --build-arg NEXT_PUBLIC_API_URL=/api so the single-origin API URL is baked into the browser bundle.
  • 3 · release — renders the infra/ecs/*.json task definitions (fills <ACCOUNT_ID>, <REGION>, <ENV>, <APP_ORIGIN> and strips the JSON comment keys), injects the freshly-pushed images, runs prisma db push as a one-off ECS task inside the VPC (RDS is private — the runner can't reach it directly), then calls aws ecs update-service for server and client with wait-for-stability + the deployment circuit breaker (auto-rollback).
Idempotent & safe to re-run Re-running on the same commit converges to the same state: terraform apply no-ops, the ECR-repo check no-ops, prisma db push only applies real schema drift, and update-service is a no-op when the task definition is unchanged. A failed rollout auto-rolls-back via the circuit breaker — it never leaves the service half-updated.

Wire it up (one-time).

  • Create a GitHub OIDC identity provider + a deploy role in IAM (trust policy scoped to this repo); put its ARN in AWS_DEPLOY_ROLE_ARN. No static AWS keys.
  • Confirm the ECS cluster + services exist (forge-<env>, forge-server-<env>, forge-client-<env>) — the pipeline updates services, it does not create them (that's Terraform / Phase 8).
  • Add the repo Secrets & Variables below, then push to main (or run it manually via workflow_dispatch).

The migration runs in the VPC, not on the runner Because RDS is not publicly accessible, the schema sync (prisma db push) is executed as a short-lived Fargate task on the private subnets using the server image — so it inherits the same DATABASE_URL/DIRECT_URL secrets. The workflow waits for it, checks the container exit code, and fails the deploy if the migration failed (before any service is updated). Seeding (npm run db:seed) is a deliberate one-off (Phase 11), not part of every deploy.
Prerequisite, not optional Phase 8's task definitions inject the /forge/<env>/SMTP_*, Google, GitHub, Discord and Groq secrets at start-up. If a referenced secret doesn't exist in Secrets Manager (Phase 3), the ECS task fails to start. Create every secret the task def lists before the first deploy.

Phase 14 · Observability, audit & email alerts

"What went wrong, when, and why" is covered at three layers, and you get an email when anything breaks. All provisioned in infra/terraform/monitoring.tf + logs.tf.

Email alerts (SNS → your inbox)

An SNS topic with an email subscription fires on the failure modes below. Set alert_email and confirm the subscription (AWS emails a one-time link after the first apply).

  • Any server error/fatal log line — a CloudWatch log-metric filter on { $.level >= 50 } (pino error=50/fatal=60). The broadest net: if the app logs an error, you hear about it.
  • ECS running tasks < desired — crash loop or a deploy that couldn't stabilise.
  • ALB 5xx spike and unhealthy targets.
  • RDS CPU high / free storage low; Redis CPU high.
  • Failed deploys — GitHub Actions emails you on any red workflow run (separate from SNS).
# terraform.tfvars
alert_email = "learnercareercouncil@nst.rishihood.edu.in"
# SMTP_FROM secret + (optional) SES verified identity use the same address
ses_identity_domain = "nst.rishihood.edu.in"
Container Insights is on The ECS cluster enables Container Insights, so per-service CPU/memory/task-count metrics (and the alarms above) are available out of the box in CloudWatch.

AWS-API audit — CloudTrail

infra/terraform/cloudtrail.tf provisions the fourth layer: who changed what in the AWS account itself (console + API). It's a multi-region trail with log-file validation, a versioned/locked-down S3 sink, and a CloudWatch Logs stream so security events alarm to the same SNS email:

  • Root account usage — any API/console call made as root.
  • Unauthorized / denied API calls — a spike of AccessDenied/UnauthorizedOperation (misconfig or probing).
  • All management events (read + write) retained in S3 for a year for forensic lookup.
Already have an org-level trail? Set enable_cloudtrail = false to skip the trail (avoids a duplicate + cost) — the bucket, CW Logs stream, and the two security alarms are created together, so disabling turns off the whole block cleanly.
