G 1 · Google OAuth — login (required)

The platform's ONLY login method. Access is granted only if the Google account is on the rishihood.edu.in hosted-domain and the email already exists in the users table.

  1. Go to the Google Cloud Console → create a project (e.g. forge).
  2. Open APIs & Services → OAuth consent screen. Choose Internal if your org is a Google Workspace for rishihood.edu.in (recommended — auto-limits to your domain). Otherwise External and we enforce the domain server-side.
  3. Add scopes: openidemailprofile
  4. Open APIs & Services → Credentials → Create credentials → OAuth client ID → Web application.
  5. Add these Authorized redirect URIs (server callback):
    Production
    Local devhttp://localhost:4000/api/auth/google/callback
  6. Copy the Client ID and Client Secret.
Env you get → server/.env
GOOGLE_OAUTH_CLIENT_ID · GOOGLE_OAUTH_CLIENT_SECRET · GOOGLE_OAUTH_REDIRECT_URI · ALLOWED_HOSTED_DOMAIN=rishihood.edu.in

⌥ 2 · GitHub — App + webhooks

Org-owned GitHub App for @lcc-ai-nst: one org installation reads all repo activity, plus a per-user OAuth that links each Forge user to their GitHub identity.

Two-token model. The App is installed once on the @lcc-ai-nst org → its installation token reads all org repo activity (commits, PRs, reviews, issues). Each person's “Connect GitHub” is only a per-user OAuth to learn their GitHub id + username, so org activity can be attributed to the right Forge user. You do not store a token per user.

→ Full org + OAuth walkthrough lives on its own page: GitHub App + OAuth Setup (create under the org, URLs, webhooks/smee, permissions, install on the org, get the Installation ID).

Env you get → server/.env
GITHUB_APP_ID · GITHUB_APP_PRIVATE_KEY · GITHUB_APP_INSTALLATION_ID · GITHUB_WEBHOOK_SECRET

D 3 · Discord — application & bot

OAuth to verify a member's username (today); a bot for activity / onboarding / announcements (later).

Minimum for username-only (today)
Only steps 1–3 below. Create the Application, generate the Client Secret, add the OAuth redirect, request scope identify. No bot, no intents, no interactions endpoint, no guild. Exchange the code → GET /users/@me → store the Discord id (snowflake, stable) + username. Env needed now: DISCORD_CLIENT_ID · DISCORD_CLIENT_SECRET.
Discord ≠ GitHub for activity. A user's OAuth token cannot read their messages/joins/reactions — identify/guilds only say who they are and which servers they're in. Capturing member activity or sending announcements requires the bot in your server listening to gateway events (live from when it joins — it does not backfill history). That's the future phase below.
  1. now Open the Discord Developer Portal → New Application (e.g. Forge). Copy the Application (Client) ID.
  2. now OAuth2 → reset secret → copy the Client Secret.
  3. now OAuth2 → Redirects, add (scope identify + optional guilds to check server membership). localhost is fine here — it's a browser redirect:
    Production
    Local devhttp://localhost:4000/api/integrations/discord/callback
  4. future Bot → Add Bot → Reset Token → copy the Bot Token (highly sensitive). Also copy the Public Key from General Information.
  5. future Under Bot → Privileged Gateway Intents: enable Server Members (for onboarding/joins). Leave Message Content off unless you must read message text.
  6. future General Information → Interactions Endpoint URL (Ed25519-verified with the Public Key). Unlike the OAuth redirect, this can't be localhost — Discord POSTs to it, so use a smee/ngrok tunnel locally:
    Interactions
  7. future Invite the bot: OAuth2 → URL Generator → scopes bot + applications.commands; minimal permissions (View Channels, Read Message History, Send Messages, Embed Links). Open the URL and add it to your server; note the Guild (Server) ID.
Env you get → server/.env
Now: DISCORD_CLIENT_ID · DISCORD_CLIENT_SECRET
Future (bot phase): DISCORD_BOT_TOKEN · DISCORD_PUBLIC_KEY · DISCORD_GUILD_ID

⌗ 4 · Google Calendar API

Sync reviews, deadlines, meetings and drive-wide events to one shared Forge calendar. Same Cloud project as Google login.

  1. Enable the API. In the same Cloud project as Google login: APIs & Services → Library → search Google Calendar API → Enable.
✅ Recommended — Option A: Service account + one shared “Forge Drive” calendar
The server uses one dedicated calendar that LCC schedules everything on (reviews, deadlines, meetings). Share that calendar with the service account once → Forge sees every event on it. No per-user login, no redirect URL, no token refresh.
How the access actually works — read this first. A service account is its own Google identity; it can only see (a) calendars explicitly shared with it and (b) events it creates. It does NOT automatically see anyone's personal calendar.
  • ✅ To see all meetings LCC schedules, LCC must put them on the one shared “Forge Drive” calendar (not their personal calendars). Then Forge sees them all.
  • ❌ Meetings on someone's personal Google calendar stay invisible — unless that person shares their calendar, or you set up domain-wide delegation (Workspace super-admin lets the SA impersonate users org-wide — heavier; only if you truly need everyone's personal calendars).
  1. Create the shared calendar. Have LCC (or you) create a Google Calendar named e.g. “Forge Drive” and make it the single place all drive meetings/reviews/deadlines are scheduled.
  2. Create the service account. APIs & Services → Credentials → Create credentials → Service account → name it forge-calendar → Create. (No roles needed — access comes from sharing the calendar in step 4.)
  3. Make a JSON key. Open the service account → Keys → Add key → Create new key → JSON → it downloads a file. That file's full contents are GOOGLE_SA_KEY. Also copy the service account's email (e.g. forge-calendar@<project>.iam.gserviceaccount.com).
  4. Share the “Forge Drive” calendar with that email → Google Calendar → that calendar → Settings → Share with specific people → Add people → paste the service-account email → choose the permission:
    If Forge…Pick
    only views LCC's meetingsSee all event details (read-only)
    also creates events (auto review/deadline entries)Make changes to events
  5. Get the Calendar ID. Same calendar's Settings → Integrate calendar → Calendar ID (e.g. c_abc123@group.calendar.google.com). That's GOOGLE_CALENDAR_ID — not primary for a shared calendar.
Env (Option A) → server/.env
GOOGLE_SA_KEY · GOOGLE_CALENDAR_ID  (no client id/secret needed)
Option B — OAuth (only if you need per-organizer calendars). Use this if events must land on each teacher's own Google calendar instead of one shared one. More moving parts: a consent screen, a redirect URL, and stored per-user tokens you must refresh.
  1. APIs & Services → Credentials → Create credentials → OAuth client ID → Web application.
  2. On the OAuth consent screen, add scope https://www.googleapis.com/auth/calendar.events.
  3. Add the Authorized redirect URIs (localhost is fine — browser redirect):
    Production
    Local devhttp://localhost:4000/api/integrations/calendar/callback
  4. Copy the Client ID + Client Secret; each organizer authorizes once. Calendar ID is each user's primary.
Env (Option B): GOOGLE_CALENDAR_CLIENT_ID · GOOGLE_CALENDAR_CLIENT_SECRET · GOOGLE_CALENDAR_ID

✉ 5 · Email — SMTP (nodemailer)

Server-side transactional email (invites, notifications, announcements) via nodemailer over SMTP. The transport degrades gracefully — if SMTP isn't set, email is logged and skipped, never crashing core flows.

✅ Google Workspace / Gmail (App Password)
Send from learnercareercouncil@nst.rishihood.edu.in via Gmail's SMTP. Requires 2-Step Verification on that account, then a 16-char App Password (the account password will not work for SMTP).
  1. Enable 2-Step Verification on the sending account: myaccount.google.com/security → 2-Step Verification → turn on. (App Passwords don't appear until this is on.)
  2. Create an App Password: myaccount.google.com/apppasswords → name it forge-smtp → Create → copy the 16-character code (shown once). That's SMTP_PASS — remove the spaces.
  3. Set the SMTP env:
    VarValue
    SMTP_HOSTsmtp.gmail.com
    SMTP_PORT587 (STARTTLS; use 465 for implicit TLS)
    SMTP_USERthe full sending address
    SMTP_PASSthe 16-char App Password
    SMTP_FROMForge <learnercareercouncil@nst.rishihood.edu.in>
  4. Restart the server — on boot it runs verifyMailer() and logs SMTP transport verified (or the error). Send a test from your code via sendEmail(...).
Limits / prod. Workspace SMTP caps at ~2,000 recipients/day — fine for invites & notices, not bulk blasts. In production prefer AWS SES (email-smtp.<region>.amazonaws.com, port 587, SES SMTP credentials) — same env vars, only host/creds change, since the nodemailer transport is generic. SES needs domain/sender verification and leaving the sandbox.
Env you get → server/.env
SMTP_HOST · SMTP_PORT · SMTP_USER · SMTP_PASS · SMTP_FROM

≡ 7 · Final server/.env

Fill the values you collected. Required to boot: DB + session + Google login. The rest enable their integrations. In production these live in AWS Secrets Manager (/forge/<env>/*), not a file.

The client holds NONE of these — only NEXT_PUBLIC_API_URL + APP_MODE. Never commit .env; rotate any key that has ever been shared.
